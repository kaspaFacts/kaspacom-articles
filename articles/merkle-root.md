## **Introductory Level - Securing the Block Body to the Header - Merkle Root and Accepted Merkle Root**

What secures the block body to the header?

The Merkle Root of the transactions in the body.

What does that mean?  This article is put together in a way that assumes no prior knowledge, so we start with a Merkle Tree.  What a Merkle tree is, how it is constructed and how it prevents tampering.  Then how it applies to both Bitcoin and Kaspa.

**Merkel Tree -** In cryptography and computer science, a Merkle Tree, also known as a Hash Tree, is a tree-like structure where each "leaf" node holds the cryptographic hash of a data block.  Non-leaf nodes, often referred to as branches or inner nodes, contain the cryptographic hash of their child nodes' labels.  This structure enables efficient and secure validation of large data sets' contents.  A Merkle Tree extends the concepts of hash lists and hash chains.

![undefined](https://cdn.buttercms.com/bTlUGaylTH6RlSxj3DkX)

**Data Block -** In a Merkle tree, a data block is a segment of raw data, such as a transaction, file chunk, or record, that forms the foundation of the tree's structure. Each data block is individually hashed using a cryptographic hash function, like SHA-256 or BLAKE2b, to produce a leaf node hash. These blocks are the core data units the Merkle tree organizes and verifies, allowing efficient integrity checks and validation across large datasets. Any modification to a data block alters its hash, propagating changes through the tree to the root hash, ensuring tamper-evident data consistency in systems like blockchains.

![merkle tree data.webp](https://cdn.buttercms.com/a03pGT4Rv2z7CWKEzZAj)
**Leaf -** In a Merkle tree, a data block is a discrete segment of information, such as a portion of a file, a transaction in a blockchain, or any structured data unit.  Each data block is processed through a cryptographic hash function, such as SHA-256 or BLAKE2b, which generates a unique, fixed-length hash value that serves as the label for a leaf node in the tree. This hash encapsulates the content of the data block, ensuring that even a minor change in the data would result in a completely different hash.  The leaf node, by storing this hash, acts as a digital fingerprint of the data block, enabling efficient verification of its integrity and authenticity within the larger Merkle tree structure.

![merkle tree leaf.webp](https://cdn.buttercms.com/NUMjXSjoTl6MaXsMDly6)

**Inner Node -** In a Merkle tree, an inner node is a parent node that aggregates the hashes of its child nodes, which can be either leaf nodes or other inner nodes. These child hashes are concatenated and processed through a cryptographic hash function, such as SHA-256 or BLAKE2b, to produce a single, fixed-length hash value that labels the inner node. This hash represents the combined integrity of all data in the subtree beneath it. Any alteration in the child nodes propagates upward, changing the inner node’s hash, ensuring efficient detection of inconsistencies across the Merkle tree structure.
![merkle tree inner node.webp](https://cdn.buttercms.com/58p9nsHSh27bUkRAzCkQ)

**Root Hash -** In a Merkle tree, the root hash is the singular hash value at the topmost node, encapsulating the entire tree’s data. It is generated by hashing the concatenated hashes of its immediate child nodes, typically inner nodes, using a cryptographic hash function like SHA-256 or BLAKE2b. The root hash serves as a compact, unique summary of all underlying data blocks, enabling rapid verification of the integrity and consistency of large datasets. Any change in a single leaf node alters the root hash, making it a critical component for data validation in systems like blockchains.

![merkle tree root.webp](https://cdn.buttercms.com/PSkxbWnsSTeVcFEWXXme)

## **Simplified Definitions**

**Merkle Tree -** A binary tree structure composed of leaf nodes, inner nodes, and a root hash, used to efficiently verify the integrity and consistency of large datasets, such as transactions in a blockchain, by organizing data blocks into a structure of cryptographic hashes.


**Data Block -** A discrete unit of information, such as a transaction or a portion of a file, that is hashed to create a leaf node’s label in a Merkle tree, forming the foundation of the tree’s structure.


**Leaf -** A node in a Merkle tree that stores the cryptographic hash of a single data block, such as a transaction or file segment, serving as a digital fingerprint to ensure the data’s integrity and authenticity.

**Inner Node -** A parent node in a Merkle tree that holds the cryptographic hash of its child nodes’ concatenated hashes, summarizing the integrity of the subtree below it and facilitating efficient verification.

**Root Hash -** The topmost node in a Merkle tree, containing a single cryptographic hash that encapsulates the hashes of all underlying nodes, providing a compact summary for verifying the entire dataset’s integrity.

A Merkle Tree is just a structure, consisting of Data and Hashes of that Data, that ensures the integrity and consistency of that Data.

## **Bitcoin and Kaspa**

**Bitcoin -** a Merkle Tree organizes transaction data within each block.  Transactions are hashed into leaf nodes, paired and hashed into inner nodes, and combined into a single root hash stored in the block header.  This structure enables efficient verification of transaction integrity and inclusion within the block.

**Kaspa -** a Merkle Tree organizes transaction data within each block.  Transactions are hashed into leaf nodes, paired and hashed into inner nodes, and combined into a single root hash stored in the block header.  This structure enables efficient verification of transaction integrity and inclusion within the block.

So what's the difference if both Bitcoin and Kaspa use a Merkle Tree?  Bitcoin allows blocks to point to only one previous block.  Kaspa allows blocks to point to multiple previous blocks, and allows parallel blocks to coexist, where order between these blocks in the [DAG](https://www.kaspa.com/learn-kaspa/post/dag-and-kaspa) cannot be known, but a Merkle Tree is order dependent, and the entire point of a Merkle Tree is to preserve order, how can we resolve this?  By adding an additional Merkle Tree.
